# create-pr

`.github/pull_request_template.md` 구조(관련 이슈 / 작업 내용 / 참고 사항 / 체크리스트)를 그대로 따라 PR 본문을 작성하고 `gh pr create --body`로 전달한다.

## 절차

1. `.github/pull_request_template.md`를 읽어 최신 템플릿 구조를 확인한다.
2. 대상 브랜치의 커밋 히스토리와 diff(`git log`, `git diff <base>...HEAD`)를 확인한다 — 마지막 커밋만이 아니라 브랜치에 포함된 모든 커밋을 근거로 삼는다.
3. 제목은 `브랜치명의 앞 키워드: 간단한 요약` 형식으로 짓는다. 예를 들어 브랜치가 `feat/dashboard`면 `feat: 대시보드 응답 시간 차트 기간별 필터링 추가`처럼 `/` 앞부분을 그대로 키워드로 쓰고, 콜론 뒤에 브랜치 전체 작업을 온전한 문장 대신 간단한 명사구로 요약한다.
4. 템플릿 섹션별로 본문을 작성한다:
   - **관련 이슈**: 사용자가 이슈 번호를 알려준 경우에만 채운다. 모르면 템플릿 형식만 남기고 비워둔다 (번호를 추측해서 채우지 않는다).
   - **작업 내용**: 커밋 히스토리와 diff를 근거로 실제 변경된 내용을 온전한 문장으로 요약한다.
   - **참고 사항**: 리뷰어가 알아야 할 맥락(브레이킹 체인지, 후속 작업 필요 여부, 임시 조치 등)이 있으면 적고 없으면 생략한다.
   - **체크리스트**: 템플릿 항목을 그대로 유지한다 (실제 확인 여부는 사람이 체크).
5. 작성한 제목/본문을 사용자에게 보여주고 확인을 받은 뒤 `gh pr create --title --body`로 실행한다. `gh pr create` 실행 자체는 원격 저장소에 영향을 주는 작업이므로 사용자 확인 없이 먼저 실행하지 않는다.

## 주의

- 이모티콘을 사용하지 않고 온전한 문장으로 작성한다 (루트 CLAUDE.md 텍스트 작성 원칙).
- push가 안 된 상태라면 먼저 push가 필요하다는 점을 사용자에게 알린다 (push도 사용자 확인이 필요한 작업).
